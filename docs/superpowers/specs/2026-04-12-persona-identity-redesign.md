# mdTI v2: 5축 조합형 성향 분류 재설계

## Context

### 문제
현재 mdTI는 두 개의 층위가 섞여 있다:
1. **수평 층위 (페르소나)**: 12가지 AI 활용 성향 분류
2. **수직 층위 (MD력)**: 0-1000점 품질 점수 + 6단계 티어 + 상위 N% 퍼센타일

MBTI에서 INTJ가 ESFP보다 "높지" 않듯이, mdTI 결과도 서열이 없어야 한다.
그러나 현재 구조에서는:
- MD력 점수가 "너 몇 점짜리야"를 말해버림
- 페르소나가 절대 점수 임계값 기반이라 암묵적 서열 존재 (minimalist < architect)
- 점수가 높은 사람만 공유 동기가 생기고, 낮은 사람은 숨김

### 목표
- **어떤 결과를 받아도 "나다!" 하고 공유하고 싶게** 만든다
- MBTI처럼 **이분법 축의 조합**으로 분류해 서열 자체를 구조적으로 불가능하게
- 정체성 + 위트로 공유 동기를 만든다

### 타깃 유저
Claude Code 및 터미널 AI 도구 사용자. 실행형 AI 시대에 반보 앞서 있는 사람들.
이 사람들에게 필요한 것은 점수가 아니라 **"나는 AI와 이렇게 일해"라는 정체성 언어**.

### 참고 자료
- Plaud 녹음 요약 (2026-04-12): "AI 사용자 유형을 '조합형 MDTI'로 재정의" — 조합형 프레임의 바이럴력 논거
- AI Native Camp 하네스 1→10 세미나: 6축 프레임워크 (`/mdti/references.md`)
- CIVC 프레임워크: Constrain / Inform / Verify / Correct

### 범위
- MD력 점수 시스템 완전 제거
- **12 고정 페르소나 → 5축 이분법 조합형 (32타입)으로 전환**
- **모듈형 콘텐츠 시스템** (5축 × 2방향 = 10블록 조합)
- 콘텐츠 톤 전환 (깎아내리기 → 정체성 80% + 위트 20%)
- 결과 페이지 UI 재구성
- 레이더 차트 → 5축 성향 지도 전환

### 범위 밖
- 7개 연속 차원의 유지 (이번에 5축 이분법으로 완전 교체)
- 기존 12 페르소나의 부분 수정 (전면 교체)

## 5축 정의

### 축 선정 기준
1. **독립성**: 서로 겹치지 않는가
2. **측정 가능**: 양쪽 다 적극적 시그널이 있는가 (부재 추론 의존 X)
3. **바이럴**: 어떤 조합이든 "나다!" 하는 스토리가 나오는가

### 대표 축: 하기스 vs 하네스

mdTI의 킥이자 대표 질문: **"너의 CLAUDE.md는 하기스야, 하네스야?"**

출처: AI Native Camp 하네스 1→10 세미나 6축 프레임워크

핵심 구분은 "만드느냐 가져다 쓰느냐"가 아니다. **하기스도 훅을 만들고, 하네스도 커뮤니티 것을 쓴다.** 진짜 차이는 **에너지의 방향**:

| | 하기스 (발산) | 하네스 (수렴) |
|---|---|---|
| 에너지 방향 | **"뭐가 있지?"** — 탐색, 확장 | **"이걸 어떻게 깎지?"** — 최적화, 정돈 |
| 훅을 만든다면 | 이것도 해보고 저것도 해보는 실험 | 검증된 패턴을 체계적으로 구축 |
| 시스템 성격 | 넓게 퍼지는 중 | 자기 방식이 수렴된 상태 |
| 둘 다 유효 | 탐색 없이는 발견이 없다 | 수렴 없이는 체계가 없다 |

### 5축 상세

| # | 축 코드 | A (첫 글자) | B (첫 글자) | 의미 | 핵심 시그널 |
|---|---------|-------------|-------------|------|------------|
| 1 | **하기스/하네스** | G (하기스) | H (하네스) | 발산 vs 수렴 | 도구/패턴 다양성·탐색 신호 vs 6축 프레임워크 일관성·최적화 신호 |
| 2 | **통제/위임** | R (Restrict) | D (Delegate) | AI 자유도 | NEVER/MUST/deny 밀도 + **permissions mode** (plan/default → R, auto/bypass → D) |
| 3 | **장황/간결** | V (Verbose) | C (Concise) | 설명 스타일 | totalLines, 단어 수, 줄당 평균 길이 |
| 4 | **설계/실행** | P (Plan) | X (eXecute) | 선후 관계 | 6축 중 구조+맥락+오케스트레이션 시그널(P) vs 검증+축적+개선 시그널(X) |
| 5 | **구조화/자유형** | S (Structured) | F (Freeform) | 문서 형태 | 헤딩 수, 리스트 비율, 계층 깊이 vs 산문체/플랫 |

### 타입 코드 예시

5글자 조합: `GRVPS`, `HDCXF` 등

| 코드 | 해석 |
|------|------|
| **GRVPS** | 하기스 + 제한 + 장황 + 설계선행 + 구조화 — "남이 만든 걸 가져다 빡빡하게 세팅해놓고 문서도 완벽한 사람" |
| **HDCXF** | 하네스 + 위임 + 간결 + 실행선행 + 자유형 — "자기 시스템 짜놓고 AI 풀어놓고 문서는 머릿속에 있는 사람" |
| **Gdcxf** | 하기스 + 위임 + 간결 + 실행선행 + 자유형 — "MCP 3개 깔고 3줄 쓰고 알아서 잘 쓰는 사람" |
| **HRVPS** | 하네스 + 제한 + 장황 + 설계선행 + 구조화 — "모든 규칙을 직접 설계하고 문서화까지 완벽한 아키텍트" |

## 축별 시그널 상세

### 축 1: 하기스(G) vs 하네스(H)

핵심: "만드느냐 vs 가져다 쓰느냐"가 아님. **에너지가 발산(탐색/확장)이냐 수렴(최적화/정돈)이냐.**
하기스도 훅을 만들고, 하네스도 커뮤니티 것을 쓴다. 시그널은 방향성을 본다.

**하기스(발산) 시그널:**
- 도구/MCP 종류가 많고 넓게 퍼짐 (다양성)
- 커뮤니티 스킬/플러그인 다수 채택
- 패턴이 여러 차원에 걸쳐 고르게 분포 (탐색 중)
- 설정이 자주 바뀌는 흔적 (실험적)
- generic한 규칙 패턴 (아직 자기 맥락에 최적화되지 않은)

**하네스(수렴) 시그널:**
- 6축 프레임워크 시그널의 **일관성** (구조/맥락/오케스트레이션/검증/축적/개선이 연결)
- 커스텀 hooks + 자기 맥락에 특화된 규칙 (generic이 아닌)
- 검증-축적-개선 루프가 보임 (순환 구조)
- 도구 수는 적을 수 있지만 깊이 사용 (수렴)
- @import, .claude/rules 등 맥락 설계가 체계적

**판정**: 시그널 다양성/분산도(발산) vs 시그널 일관성/깊이(수렴)
**경계**: 양쪽 시그널이 비슷하면 → 전체 패턴 분산도 기반 이분

### 축 2: 통제(R) vs 위임(D)

**통제 시그널:**
- NEVER/MUST/ALWAYS/IMPORTANT 키워드 빈도
- deny rules (settings.json)
- PreToolUse hooks (차단/검증)
- permissions mode: `plan`, `default`
- 가드레일 패턴 ("~하지 마라", "절대 ~하지 않는다")

**위임 시그널:**
- permissions mode: `auto`, `bypassPermissions`
- 규칙 부재가 아닌 **명시적 위임** ("AI 판단에 맡긴다", "알아서 해")
- 최소한의 제약 + 높은 자동화
- PostToolUse hooks (사후 확인만)

**판정**: 통제 시그널 밀도 (통제 시그널 / 전체 줄 수) 기반 이분

### 축 3: 장황(V) vs 간결(C)

**판정**: totalLines 기반 이분
- 임계값: 중앙값 또는 고정값 (예: 30줄 기준)
- 확장 입력(B-path)의 경우 별도 임계값

### 축 4: 설계(P) vs 실행(X)

6축 프레임워크에서 도출:
- **설계(P) 시그널**: 구조(Scaffolding) + 맥락(Context) + 계획·실행(Orchestration) 관련 패턴
  - @imports, AGENTS.md, 프로젝트 구조 정의, 워크플로우 사전 설계
- **실행(X) 시그널**: 검증(Verification) + 축적(Compounding) + 개선(Simplification) 관련 패턴
  - verify/test 패턴, memory/compound, simplify/refactor 패턴

**판정**: P 시그널 vs X 시그널 비율

### 축 5: 구조화(S) vs 자유형(F)

**구조화 시그널:**
- 헤딩 수 (h1, h2, h3)
- 리스트 비율 (20%+ → 구조화)
- 계층 깊이 (h1 > h2 > h3 존재)
- 섹션 분리 (`---` 등)

**자유형 시그널:**
- 헤딩 0~1개
- 산문체 비율 높음
- 플랫 구조 (계층 없음)

**판정**: 헤딩 수 + 리스트 비율 기반 이분

## 콘텐츠 시스템: 고유 페르소나 + 모듈 위트

### 원리

MBTI에서 INTJ가 재밌는 건 "I+N+T+J"여서가 아니라 **"전략가"라는 캐릭터** 때문이다.
mdTI도 동일하게: 각 조합이 **축의 체크리스트가 아니라 하나의 캐릭터**로 느껴져야 한다.

### 이중 구조

| 요소 | 방식 | 이유 |
|------|------|------|
| **이름 + 한줄 + 캐릭터 서사** | 32개 **고유** | 축 간 상호작용이 만드는 캐릭터 = 공유 동기의 핵심 |
| **위트** ("~한 적 없나요?") | 축별 **모듈** (10블록) | 축별 찌르기는 모듈로 충분 |
| **탐험 제안** | 축별 **모듈** (10블록) | 반대 축 시도 유도는 모듈로 충분 |

### 32개 고유 페르소나 (이름 + 한줄 + 캐릭터 서사)

각 조합마다:
- **이름** (2-3단어): 기억하기 쉽고 자랑하고 싶은
- **한줄** (1문장): 정체성 요약
- **캐릭터 서사** (2-3문장): 축 간 **긴장/시너지**를 엮은 통합 서술

예시:

**카오스 엔지니어** (하네스·위임·간결·실행·자유형)
> "시키면 끝나 있는 시스템의 주인"
>
> 당신의 시스템은 정교하게 돌아가는데, 그걸 설명하는 문서는 어디에도 없다. 구조는 머릿속에, 실행은 AI에게, 결과는 알아서 나온다. 주변에서는 대체 어떻게 하는 건지 모르지만 — 잘 된다.

**결계 아키텍트** (하네스·통제·장황·설계·구조화)
> "빈틈없는 시스템의 설계자"
>
> 모든 규칙을 직접 설계하고, 모든 예외를 문서화하고, AI가 넘지 못할 선을 정확히 긋는다. 당신의 CLAUDE.md를 보면 하나의 완성된 운영 매뉴얼이다. 빡빡하지만 — 그래서 신뢰할 수 있다.

**직감의 서퍼** (하기스·위임·간결·실행·자유형)
> "가볍게 타지만 넘어지지 않는 사람"
>
> MCP 몇 개면 충분하고, 규칙은 최소한으로, 문서는 3줄이면 된다. AI를 믿고 파도를 타듯이 쓴다. 가벼워 보이지만 — 자기만의 감이 있다.

**매뉴얼 콜렉터** (하기스·통제·장황·설계·구조화)
> "세상의 모든 가이드를 정리해둔 사람"
>
> 남이 만든 좋은 것을 찾아내고, 빠짐없이 세팅하고, 완벽하게 문서화한다. 직접 만들지 않았을 뿐 — 당신의 정리 능력 자체가 이미 하나의 시스템이다.

(나머지 28개는 구현 단계에서 작성, 전체 목록은 별도 콘텐츠 문서)

### 축별 모듈 블록 (위트 + 탐험)

5축 × 2방향 = **10개 블록**. 위트와 탐험 제안만 담당.

**위트 블록 (기본 포맷: "~한 적 없나요?"):**

| 축 | A방향 | B방향 |
|----|-------|-------|
| 하기스/하네스 | "이번 주에 깐 MCP가 지난 주에 깐 것보다 많은 적 없나요?" | "설정 다듬다가 하루가 간 적 없나요?" |
| 통제/위임 | "NEVER 12개 써놓고도 불안해서 한 번 더 확인하러 돌아온 적 없나요?" | "bypass 모드 켜놓고 커피 마시러 갔다가, 뭔가 잘못된 적 없나요?" |
| 장황/간결 | "CLAUDE.md 수정하다가 이게 문서인지 자서전인지 헷갈린 적 없나요?" | "3줄 쓰고 '이 정도면 충분하지' 했는데, 클로드가 전혀 다른 걸 해온 적 없나요?" |
| 설계/실행 | "@import 구조 잡다가 정작 코드는 한 줄도 안 쓴 채로 하루가 간 적 없나요?" | "일단 돌려보고 고치자 했는데, 고칠 게 산더미가 된 적 없나요?" |
| 구조화/자유형 | "헤딩 레벨 정하다가 30분 쓴 적 없나요?" | "나중에 정리해야지 하고 3개월째 산문체인 적 없나요?" |

**탐험 블록:**

| 축 | A방향 탐험 | B방향 탐험 |
|----|-----------|-----------|
| 하기스/하네스 | "자주 쓰는 것 하나만 깊이 파보면 새로운 세계가 열릴 수도" | "가끔은 새로운 도구를 아무 계획 없이 깔아보는 것도 발견이야" |
| 통제/위임 | "한 번쯤 규칙 없이 AI를 풀어보면 의외의 결과가 나올지도" | "중요한 작업에 가드레일 하나만 걸어보면 밤잠이 편할 수도" |

## 성향 지도 (5축 레이더)

### 현재
- 7축, 0-100 숫자 라벨 표시

### 변경

- **숫자 없음** — 각 축의 방향만 표시 (A쪽 vs B쪽으로 어디에 기울어져 있나)
- **축 라벨**: 한글 (하기스↔하네스, 통제↔위임, 장황↔간결, 설계↔실행, 구조화↔자유형)
- **시각화 방식**: 중심선 기준 각 축의 기울기 (A쪽이면 바깥, B쪽이면 안쪽 — 또는 양방향 레이더)
- 데이터 소스: 각 축의 A/B 시그널 비율

## 콘텐츠 톤 가이드

### 정체성 블록
- **톤**: 자부심을 주는 서술. "너는 이런 사람이야"
- **원칙**: 어떤 방향이든 읽으면 "맞아, 나 이래" 하고 고개 끄덕이게
- **금지**: "잘하고 있다", "높은 수준", "뛰어난" 같은 평가형 표현
- **허용**: "이런 사람은 드물다", "이게 쉬운 일이 아닌데" 같은 희소성 표현

### 위트 블록
- **톤**: "ㅋㅋ 들켰다" — 가볍게 찌르지만 상처는 안 되게
- **기본 포맷**: **"~한 적 없나요?"** — 읽는 사람의 **경험을 소환**시키는 질문형
  - 행동만 지적하지 않고 **행동 + 결과**를 함께 제시
  - 단정("있죠?")이 아닌 **부드러운 질문**("없나요?")
  - 예: "bypass 모드 켜놓고 커피 마시러 갔다가, 뭔가 잘못된 적 없나요?"
- **원칙**: 행동 패턴을 꼬집되, 능력을 깎아내리지 않음
- **금지**: "부족하다", "못한다", "아직 멀었다" 같은 부정 평가

### 탐험 제안 블록
- **톤**: "이것도 해보면 재밌을 걸" — 반대 축 시도 유도
- **원칙**: "처방"이 아니라 "탐험". 호기심 자극
- **금지**: "개선", "보완", "강화" 같은 부족함 전제 표현

## 다음 페이즈 고려사항 (GTM 전략)

### 핵심 인사이트
- **정체성이 곧 온보딩**: "나 카오스 엔지니어야"가 AI 사용법의 언어를 준다 → 최저비용 사용자 획득
- **테스트 결과 = 세그멘테이션 데이터**: 5축 32타입은 실제 행동 기반 세그먼트. 추천 엔진의 기초
- **"더 잘 쓰게 되었다"가 리텐션**: 재측정 → 축 변화 시각화 → 성장 경로 → 지속 사용

### GTM 플라이휠
```
바이럴 진입 → 정체성 확인 → 액셔너블 제안 → 원클릭 실행 → 재측정 → 변화 공유 → 바이럴
```
**현재 빠진 고리: 제안→실행.** v2에서 최소한 "추천 세팅 5개" 수준의 액션은 붙여야 함.

### 확장 방향
1. **타입별 추천 세팅 팩** (v2 동시 출시 권장): 결과 하단 "시급 세팅 5가지" + 원클릭 복사/CLI 명령어
2. **성장 경로 대시보드**: 재분석 시 이전 결과와 비교, 축 변화 타임라인 시각화
3. **하네스 마켓플레이스 입구**: mdTI를 GTM 퍼널로, "당신 타입에 맞는 하네스" 연결 → 장기 비전
4. **팀/조직 분석**: "우리 팀 5축 분포도" → B2B 확장 경로
