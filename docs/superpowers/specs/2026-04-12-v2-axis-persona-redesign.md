## 1. 5축 재정의

### 변경 요약

| # | 축 코드 | A | B | 기존 의미 | **재정의** | 사용자 노출 별명 |
|---|---------|---|---|-----------|-----------|-----------------|
| 1 | harness | G (하기스) | H (하네스) | 탐색/구축 | **유지** — 발산(탐색·확장) vs 수렴(최적화·정돈) | 하기스 / 하네스 |
| 2 | control | R (Restrict) | D (Delegate) | 통제/위임 | **유지** — AI 자유도 | 가드레일러 / 방목러 |
| 3 | verbose | V | C | 장황/간결 (글 길이) | **리프레이밍** → 맥락/핵심 (커뮤니케이션 철학) | 내러티브 / 다이렉트 |
| 4 | plan | P (Plan) | X (eXecute) | 설계/실행 | **유지** — 선후 관계 | 청사진 / 스프린트 |
| 5 | structure | S (Structured) | F (Freeform) | _(없음)_ | **신규 도입** — 문서 형태 (체계/자유) | 아키텍트 / 프리스타일 |

### 축 3 리프레이밍 상세: 장황/간결 → 맥락/핵심

**기존 문제**: "글이 긴가 짧은가"는 단순 행동 패턴이라 정체성이 안 됨.

**리프레이밍**:

| | V = 맥락형 (Contextual) | C = 핵심형 (Essential) |
|---|---|---|
| 철학 | "AI가 **왜**를 알면 더 잘한다" | "AI는 **뭘** 할지만 알면 된다" |
| 행동 | 배경, 이유, 맥락을 충분히 설명 | 목표와 조건만 짧게 전달 |
| 강점 | AI가 의도를 깊이 이해, 맥락에 맞는 결과 | 효율적, 빠른 피드백 루프 |
| 약점 | 설명하다가 본인이 먼저 답 찾을 때도 있음 | 가끔 AI가 의도를 오해 |
| 측정 | totalLines, 단어수, **맥락 키워드** (because, 이유, 배경, context) | 짧은 문서 + 지시형 키워드 (do, run, make) |

코드 문자(V/C)는 유지. 내부 의미와 사용자 노출 프레이밍만 변경.

### 축 5 신규 도입: 구조화(S) vs 자유형(F)

| | S = 구조화 (Structured) | F = 자유형 (Freeform) |
|---|---|---|
| 문서 형태 | 헤딩, 리스트, 계층 구조, 섹션 분리 | 산문체, 플랫, 대화체 |
| 철학 | "정돈된 구조가 명확한 소통" | "자연스러운 흐름이 더 솔직한 소통" |
| 시그널 | 마크다운 헤딩 수, 리스트 비율, 계층 깊이, YAML/TOML 구조 | 산문 비율, 코드 블록 없음, 플랫 텍스트 |

## 2. 타입 코드 체계

5글자: `[G|H][R|D][V|C][P|X][S|F]` = 2^5 = **32타입**

예시:
- **GRVPS**: 하기스 + 가드레일러 + 내러티브 + 청사진 + 아키텍트
- **HDCXF**: 하네스 + 방목러 + 다이렉트 + 스프린트 + 프리스타일

## 3. 32개 페르소나

### 네이밍 원칙

1. **"나는 ○○다"라고 SNS에 쓸 때 멋있어야 한다**
2. 기능 설명이 아닌 **캐릭터/은유** — 이름만으로 이미지가 그려질 것
3. 톤 혼합: 판타지/은유 + 직업/역할 + 캐주얼/밈 + SF/테크
4. 2~5음절, 기억하기 쉽고 발음하기 좋은 이름
