## 축 1: 하기스(G) vs 하네스(H) — 발산 vs 수렴

### 축 의미
- **하기스(G)**: 에너지가 바깥으로 — 탐색, 확장, 다양한 시도
- **하네스(H)**: 에너지가 안쪽으로 — 최적화, 정돈, 자기 방식 수렴

### 기존 문제
"MCP 수 = 탐색, 훅 수 = 구축" → **도구 보유량을 세는 것이지 에너지 방향을 보는 것이 아님.**
MCP 5개를 깊이 쓰는 사람은 하네스, 훅 3개를 실험 중인 사람은 하기스일 수 있음.

### 재설계 시그널

**하기스(G) — 넓게 퍼져 있는가:**
| 시그널 | 측정 방법 | 가중치 |
|--------|-----------|--------|
| 도구 카테고리 분산 | toolNames가 3개 이상 카테고리에 걸침 (dev/comm/design/infra/data 등) | +2 |
| MCP 다양성 | mcpServerCount ≥ 3 AND 서로 다른 카테고리 | +1 |
| 플러그인 탐색 흔적 | pluginCount ≥ 3 AND pluginEnabledRatio < 0.7 (깔아보고 안 쓰는 게 있음) | +2 |
| 범용 규칙 | CLAUDE.md에 프로젝트-특화되지 않은 일반 규칙 비율 높음 | +1 |
| 도구 언급 빈도 | toolDiversity 패턴 매치 종류 ≥ 5 | +1 |

**하네스(H) — 깊이 파고 있는가:**
| 시그널 | 측정 방법 | 가중치 |
|--------|-----------|--------|
| 커스텀 제작물 | userSkillCount ≥ 1 OR userAgentCount ≥ 1 OR commandCount ≥ 2 | +2 |
| 정교한 훅 | hookPromptCount ≥ 1 (AI 판단 훅 = 정교하게 설계된 자동화) | +2 |
| 프로젝트별 맞춤 | projectMdCount ≥ 2 (프로젝트마다 다른 설정) | +1 |
| 플러그인 정제 | pluginEnabledRatio > 0.8 (쓸 것만 골라 쓰는 중) | +1 |
| 프로젝트-특화 규칙 | CLAUDE.md에 특정 프로젝트/도메인 맞춤 규칙 존재 | +1 |

### 핵심 변화
**"뭘 가지고 있나" → "어떻게 쓰고 있나"**
- 같은 MCP 5개라도: 카테고리가 다양하면 G, 한 분야에 집중이면 H
- 같은 훅 3개라도: 직접 만든 정교한 훅이면 H, 커뮤니티에서 가져온 거면 G

## 축 2: 통제(R) vs 위임(D) — AI 자유도

### 축 의미
- **통제(R)**: AI에게 빡빡 — 규칙, 제약, 가드레일
- **위임(D)**: AI를 믿고 맡김 — 자유, 판단 위임

### 기존 문제
R 시그널 28개 이상 vs D 시그널 1~2개. D가 "적극적 위임"이 아니라 "아무것도 안 함"으로 측정됨.

### 재설계 시그널

**통제(R):**
| 시그널 | 측정 방법 | 가중치 |
|--------|-----------|--------|
| deny 규칙 | denyCount ≥ 1 | +1 (≥3이면 +2) |
| 제약 키워드 | NEVER, MUST, 금지, CRITICAL, ALWAYS, IMPORTANT 매치 수 | 종류당 +1 |
| 위험 차단 | blocksDangerousOps = true | +1 |
| plan 모드 | defaultMode: "plan" 또는 "default" | +1 |
| 보안 규칙 | .env, token, secret 관련 제약 패턴 | 종류당 +1 |
| 행동 제약 밀도 | (제약 키워드 수 / totalLines) 높음 | +1 |

**위임(D) — 보강:**
| 시그널 | 측정 방법 | 가중치 |
|--------|-----------|--------|
| auto 모드 | defaultMode: "auto" | +2 |
| bypass 설정 | bypassPermissions = true | +1 |
| AI 판단 훅 비율 | hookPromptCount > hookCommandCount (AI한테 판단 맡김) | +1 |
| 위임 키워드 | "알아서", "자유롭게", "판단에 맡", "스스로 판단", "자율" | 종류당 +1 |
| 낮은 제약 밀도 | denyCount ≤ 1 AND hookCount ≥ 1 (훅은 쓰면서 규칙은 안 건 = 신뢰) | +1 |
| 소극적 위임 | ruleCount < 5 AND totalLines > 30 (글은 쓰되 규칙은 적음) | +1 |

### 핵심 변화
- D에 **적극적 위임 시그널** 추가 (위임 키워드, AI 판단 훅 비율)
- D에 **의도적 비제약 시그널** 추가 (훅은 있는데 규칙은 안 건 = 의도적 자유)

## 축 3: 맥락(V) vs 핵심(C) — 커뮤니케이션 철학

### 축 의미 (리프레이밍)
- **맥락(V)**: "AI가 **왜**를 알면 더 잘한다" — 배경, 이유, 맥락을 설명
- **핵심(C)**: "AI는 **뭘** 할지만 알면 된다" — 목표와 조건만 전달

### 기존 문제
`claudeMdLines > 100이면 V, 아니면 C` — 줄 수 하나로만 판정. 리프레이밍된 의미를 전혀 반영 못 함.

### 재설계 시그널

**맥락(V):**
| 시그널 | 측정 방법 | 가중치 |
|--------|-----------|--------|
| 맥락 키워드 | "때문에", "이유는", "배경", "왜냐", "~이므로", "~기 위해", "~하려면", "context", "참고로" | 종류당 +1 |
| 설명형 문장 | 규칙 뒤에 이유가 붙은 패턴 ("~해줘. 왜냐하면~", "~금지 — ~때문") | +2 |
| 줄 수 (보조) | claudeMdLines > 100 | +1 |
| 서술 비율 | 규칙당 평균 줄 수 > 2 (한 규칙에 설명이 같이 있음) | +1 |

**핵심(C):**
| 시그널 | 측정 방법 | 가중치 |
|--------|-----------|--------|
| 명령형 밀도 | "해줘", "하지 마", "쓰지 마", "금지", "사용", "실행" — 설명 없는 짧은 지시문 비율 | +1 |
| 단문 규칙 | 불릿 포인트 규칙 수 AND 규칙당 평균 줄 수 ≤ 1 (설명 없이 나열) | +2 |
| 줄 수 (보조) | claudeMdLines ≤ 50 | +1 |
| 맥락 키워드 부재 | 위 맥락 키워드가 0개 | +1 |

### 핵심 변화
**"글이 긴가 짧은가" → "왜를 말하는가 뭘만 말하는가"**
- 줄 수는 보조 시그널로만 사용
- 핵심은 **맥락 키워드 존재 여부** + **규칙에 이유가 붙어있는가**

## 축 4: 구조화(S) vs 자유형(F) — 문서 형태

### 축 의미
- **구조화(S)**: 체계적 문서 — 헤딩, 리스트, 계층 구조
- **자유형(F)**: 자연스러운 문서 — 산문, 대화체, 플랫

### 기존 코드
`judgeStructureAxis()` 함수 존재 — `headingCount ≥ 3 OR listRatio ≥ 0.2`

### 재설계 시그널

**구조화(S):**
| 시그널 | 측정 방법 | 가중치 |
|--------|-----------|--------|
| 헤딩 | 마크다운 헤딩(#) ≥ 3개 | +2 |
| 리스트 비율 | 불릿/번호 리스트 비율 ≥ 20% | +1 |
| 계층 깊이 | 헤딩 레벨이 2단계 이상 (## + ### 등) | +1 |
| 섹션 분리 | 빈 줄 또는 구분선(---)으로 섹션 나뉨 | +1 |

**자유형(F):**
| 시그널 | 측정 방법 | 가중치 |
|--------|-----------|--------|
| 산문 비율 | 헤딩 0~1개 AND 리스트 비율 < 10% | +2 |
| 대화체 | 평서문/구어체 패턴 ("~하면 돼", "~인데", "~거든") | +1 |
| 플랫 구조 | 헤딩 레벨이 1단계만 (# 또는 ##만) | +1 |

### 핵심 변화
- 기존 코드 보강 (계층 깊이, 섹션 분리 추가)
- F 시그널에 대화체 패턴 추가

## 시그널 독립성 검증

| | G/H | R/D | V/C | S/F |
|---|---|---|---|---|
| **G/H** | — | 독립 (도구 사용 패턴 vs 제약 수준) | 독립 (도구 패턴 vs 설명 스타일) | 독립 (도구 패턴 vs 문서 형태) |
| **R/D** | | — | ⚠️ 약한 상관 (R이면 V일 가능성?) → 규칙 키워드 vs 맥락 키워드 분리로 해결 | 독립 (제약 수준 vs 문서 형태) |
| **V/C** | | | — | ⚠️ 약한 상관 (V+S 자주 나올 수 있음) → 줄 수/키워드 vs 구조 분석 분리로 해결 |
| **S/F** | | | | — |

주요 겹침 위험:
1. R(통제) ↔ V(맥락): "NEVER ~해. 왜냐하면 ~" → R 키워드(NEVER)와 V 키워드(왜냐하면)를 별도 카운트하면 분리 가능
2. V(맥락) ↔ S(구조): 길고 구조적인 문서 → V는 "이유 키워드 유무", S는 "포맷(헤딩/리스트)" → 측정 대상이 다르므로 분리 가능
